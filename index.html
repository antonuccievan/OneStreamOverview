    Dimensions and the POV

    Core concept overview
    • Dimensions + Metadata define reporting structure
    • Dimensions are ways to slice data: e.g., Entity, Account, Cost Center

    Dimensions & Metadata — Details

    • Multi-dimensional model; metadata composes dimensions
    • 18 dimensions in OneStream (plus cube)
    • Each amount is identified by a 19-member "address"

    Hierarchy & Members

    • Members arranged in hierarchies (parent, child, descendant, base-level)
    • Members can appear multiple times via Relationships
    • Aggregation & consolidation use these relationships

    POV (Point of View)

    • POV determines which slice of the cube you are viewing
    • If data is missing or unexpected, check the POV (Cube POV, Workflow POV, Global)

    Cube

    • A cube organizes dimensions, rules, and data
    • Multiple cubes can be built and linked for specific solutions

    Entity

    • Represents legal entity or business unit
    • Currency translation and consolidation governed by Entity
    • Data consolidates up the Entity hierarchy

    Parent

    • Parent of a selected Entity
    • Entities may appear under different parents across hierarchies (different currency/ownership)

    Consolidation

    • Tracks aggregation during consolidation
    • Consolidation members and sequence drive how amounts roll up

    Consolidation — Top Hierarchy

    • Local: uses functional currency assigned to Entity
    • Translated: Local translated to parent currency
    • OwnerPreAdj, Share (ownership), Elimination, OwnerPostAdj; Top = Share + Eliminations + OwnerPostAdj

    Consolidation Sequence

    Reference OneStream Design & Reference Guide for full sequence
    • Follow documented consolidation sequence for accurate results

    Consolidation — Currency Hierarchy

    • Currencies enabled per reporting requirements
    • Each Entity has a default currency; translations occur automatically
    • System members such as Local and Translated are dynamic

    Consolidation — Analysis Hierarchy

    • Aggregated member for planning/forecasting (simplified aggregation; no eliminations)
    • Used to accelerate performance for planning scenarios

    Scenario

    • Typical Scenarios: Actuals, Budget, Forecast, etc.

    Scenario Details

    • Scenarios do not aggregate across certain dimensions by design; they are configured per requirement
    • Scenario Types group scenarios to vary rules and behavior

    Time (Calendar)

    • Standard calendars: 12 months, 4 quarters, 2 half years, 1 year
    • Examples: 2021M1, 2021Q1; fiscal vs calendar options; weekly apps possible

    View

    • Built-in calendar intelligence for flow-type accounts (MTD, QTD, YTD)
    • Members for annotations, calc status, and predefined math (TrailingXMonths Sum/Average)

    Account

    • Accounts dimension = Chart of Accounts
    • Accounts have types (revenue, expense, asset, liability) that affect aggregation

    Flow

    • Visibility into account movements (roll forwards)
    • Supports FX by account, cash flow reporting, historical-rate overrides

    Flow — Details

    • Many calculated members; simple for P&L, more extensive for Balance Sheet

    Origin

    • Tracks input source/origination of data — important for workflow and data lineage

    Origin Members

    • Import: file loads via Workflow
    • Forms: data via Forms or Excel templates
    • AdjInput, AdjConsolidated, Elimination, BeforeAdj (Import + Forms = BeforeAdj)

    ICP (Intercompany Partner)

    • Indicates partner Entity for a transaction (members flagged IsIC)
    • Key for intercompany automation and eliminations

    User-Defined Dimensions (UD1–UD8)

    • Eight UDs available (UD1–UD8)
    • Riveron common uses: UD6 reporting, UD7 tracking, UD8 dynamic calculations; others configurable per client
    • UDs can be constrained to account groups

    Extensibility

    • Metadata can vary in level of detail across contexts (Extensible Dimensionality)
    • Design & Reference Guide provides recommended patterns

    Extensibility Notes

    • Use different levels of granularity for corporate vs actual vs budget models where appropriate
